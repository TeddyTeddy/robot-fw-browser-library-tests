48USAGE = """Rebot -- Robot Framework report and log generator 

49 

50Version: <VERSION> 

51 

52Usage: rebot [options] robot_outputs 

53 or: python -m robot.rebot [options] robot_outputs 

54 or: python path/to/robot/rebot.py [options] robot_outputs 

55 or: java -jar robotframework.jar rebot [options] robot_outputs 

56 

57Rebot can be used to generate logs and reports in HTML format. It can also 

58produce new XML output files which can be further processed with Rebot or 

59other tools. 

60 

61The easiest way to execute Rebot is using the `rebot` command created as part 

62of the normal installation. Alternatively it is possible to execute the 

63`robot.rebot` module directly using `python -m robot.rebot`, where `python` 

64can be replaced with any supported Python interpreter like `jython`, `ipy` or 

65`python3`. Yet another alternative is running the `robot/rebot.py` script like 

66`python path/to/robot/rebot.py`. Finally, there is a standalone JAR 

67distribution available. 

68 

69Inputs to Rebot are XML output files generated by Robot Framework or by earlier 

70Rebot executions. When more than one input file is given, a new top level test 

71suite containing suites in the given files is created by default. This allows 

72combining multiple outputs together to create higher level reports. An 

73exception is that if --merge is used, results are combined by adding suites 

74and tests in subsequent outputs into the first suite structure. If same test 

75is found from multiple outputs, the last one replaces the earlier ones. 

76 

77For more information about Rebot and other built-in tools, see 

78http://robotframework.org/robotframework/#built-in-tools. For more details 

79about Robot Framework in general, go to http://robotframework.org. 

80 

81Options 

82======= 

83 

84 --rpa Turn on the generic automation mode. Mainly affects 

85 terminology so that "test" is replaced with "task" 

86 in logs and reports. By default the mode is got 

87 from the processed output files. New in RF 3.1. 

88 -R --merge When combining results, merge outputs together 

89 instead of putting them under a new top level suite. 

90 Example: rebot --merge orig.xml rerun.xml 

91 -N --name name Set the name of the top level suite. 

92 -D --doc documentation Set the documentation of the top level suite. 

93 Simple formatting is supported (e.g. *bold*). If 

94 the documentation contains spaces, it must be quoted. 

95 Example: --doc "Very *good* example" 

96 -M --metadata name:value * Set metadata of the top level suite. Value can 

97 contain formatting similarly as --doc. 

98 Example: --metadata Version:1.2 

99 -G --settag tag * Sets given tag(s) to all tests. 

100 -t --test name * Select tests by name or by long name containing also 

101 parent suite name like `Parent.Test`. Name is case 

102 and space insensitive and it can also be a simple 

103 pattern where `*` matches anything, `?` matches any 

104 single character, and `[chars]` matches one character 

105 in brackets. 

106 --task name * Alias to --test. Especially applicable with --rpa. 

107 -s --suite name * Select suites by name. When this option is used with 

108 --test, --include or --exclude, only tests in 

109 matching suites and also matching other filtering 

110 criteria are selected. Name can be a simple pattern 

111 similarly as with --test and it can contain parent 

112 name separated with a dot. For example, `-s X.Y` 

113 selects suite `Y` only if its parent is `X`. 

114 -i --include tag * Select tests by tag. Similarly as name with --test, 

115 tag is case and space insensitive and it is possible 

116 to use patterns with `*`, `?` and `[]` as wildcards. 

117 Tags and patterns can also be combined together with 

118 `AND`, `OR`, and `NOT` operators. 

119 Examples: --include foo --include bar* 

120 --include fooANDbar* 

121 -e --exclude tag * Specify tests not to be included by tag. They are not 

122 selected even if included with --include. Tags are 

123 matched using same rules as with --include. 

124 --processemptysuite Processes output also if the top level suite is 

125 empty. Useful e.g. with --include/--exclude when it 

126 is not an error that there are no matches. 

127 -c --critical tag * Deprecated since RF 4.0 and has no effect anymore. 

128 -n --noncritical tag * Deprecated since RF 4.0 and has no effect anymore. 

129 Use --skiponfailure when starting execution instead. 

130 -d --outputdir dir Where to create output files. The default is the 

131 directory where Rebot is run from and the given path 

132 is considered relative to that unless it is absolute. 

133 -o --output file XML output file. Not created unless this option is 

134 specified. Given path, similarly as paths given to 

135 --log, --report and --xunit, is relative to 

136 --outputdir unless given as an absolute path. 

137 -l --log file HTML log file. Can be disabled by giving a special 

138 name `NONE`. Default: log.html 

139 Examples: `--log mylog.html`, `-l none` 

140 -r --report file HTML report file. Can be disabled with `NONE` 

141 similarly as --log. Default: report.html 

142 -x --xunit file xUnit compatible result file. Not created unless this 

143 option is specified. 

144 --xunitskipnoncritical Deprecated since RF 4.0 and has no effect anymore. 

145 -T --timestampoutputs When this option is used, timestamp in a format 

146 `YYYYMMDD-hhmmss` is added to all generated output 

147 files between their basename and extension. For 

148 example `-T -o output.xml -r report.html -l none` 

149 creates files like `output-20070503-154410.xml` and 

150 `report-20070503-154410.html`. 

151 --splitlog Split the log file into smaller pieces that open in 

152 browsers transparently. 

153 --logtitle title Title for the generated log file. The default title 

154 is `<SuiteName> Test Log`. 

155 --reporttitle title Title for the generated report file. The default 

156 title is `<SuiteName> Test Report`. 

157 --reportbackground colors Background colors to use in the report file. 

158 Either `all_passed:critical_passed:failed` or 

159 `passed:failed`. Both color names and codes work. 

160 Examples: --reportbackground green:yellow:red 

161 --reportbackground #00E:#E00 

162 -L --loglevel level Threshold for selecting messages. Available levels: 

163 TRACE (default), DEBUG, INFO, WARN, NONE (no msgs). 

164 Use syntax `LOGLEVEL:DEFAULT` to define the default 

165 visible log level in log files. 

166 Examples: --loglevel DEBUG 

167 --loglevel DEBUG:INFO 

168 --suitestatlevel level How many levels to show in `Statistics by Suite` 

169 in log and report. By default all suite levels are 

170 shown. Example: --suitestatlevel 3 

171 --tagstatinclude tag * Include only matching tags in `Statistics by Tag` 

172 in log and report. By default all tags are shown. 

173 Given tag can be a pattern like with --include. 

174 --tagstatexclude tag * Exclude matching tags from `Statistics by Tag`. 

175 This option can be used with --tagstatinclude 

176 similarly as --exclude is used with --include. 

177 --tagstatcombine tags:name * Create combined statistics based on tags. 

178 These statistics are added into `Statistics by Tag`. 

179 If the optional `name` is not given, name of the 

180 combined tag is got from the specified tags. Tags are 

181 matched using the same rules as with --include. 

182 Examples: --tagstatcombine requirement-* 

183 --tagstatcombine tag1ANDtag2:My_name 

184 --tagdoc pattern:doc * Add documentation to tags matching the given 

185 pattern. Documentation is shown in `Test Details` and 

186 also as a tooltip in `Statistics by Tag`. Pattern can 

187 use `*`, `?` and `[]` as wildcards like --test. 

188 Documentation can contain formatting like --doc. 

189 Examples: --tagdoc mytag:Example 

190 --tagdoc "owner-*:Original author" 

191 --tagstatlink pattern:link:title * Add external links into `Statistics by 

192 Tag`. Pattern can use `*`, `?` and `[]` as wildcards 

193 like --test. Characters matching to `*` and `?` 

194 wildcards can be used in link and title with syntax 

195 %N, where N is index of the match (starting from 1). 

196 Examples: --tagstatlink mytag:http://my.domain:Title 

197 --tagstatlink "bug-*:http://url/id=%1:Issue Tracker" 

198 --expandkeywords name:<pattern>|tag:<pattern> * 

199 Matching keywords will be automatically expanded in 

200 the log file. Matching against keyword name or tags 

201 work using same rules as with --removekeywords. 

202 Examples: --expandkeywords name:BuiltIn.Log 

203 --expandkeywords tag:expand 

204 New in RF 3.2. 

205 --removekeywords all|passed|for|wuks|name:<pattern>|tag:<pattern> * 

206 Remove keyword data from all generated outputs. 

207 Keywords containing warnings are not removed except 

208 in the `all` mode. 

209 all: remove data from all keywords 

210 passed: remove data only from keywords in passed 

211 test cases and suites 

212 for: remove passed iterations from for loops 

213 wuks: remove all but the last failing keyword 

214 inside `BuiltIn.Wait Until Keyword Succeeds` 

215 name:<pattern>: remove data from keywords that match 

216 the given pattern. The pattern is matched 

217 against the full name of the keyword (e.g. 

218 'MyLib.Keyword', 'resource.Second Keyword'), 

219 is case, space, and underscore insensitive, 

220 and may contain `*`, `?` and `[]` wildcards. 

221 Examples: --removekeywords name:Lib.HugeKw 

222 --removekeywords name:myresource.* 

223 tag:<pattern>: remove data from keywords that match 

224 the given pattern. Tags are case and space 

225 insensitive and patterns can contain `*`, 

226 `?` and `[]` wildcards. Tags and patterns 

227 can also be combined together with `AND`, 

228 `OR`, and `NOT` operators. 

229 Examples: --removekeywords foo 

230 --removekeywords fooANDbar* 

231 --flattenkeywords for|foritem|name:<pattern>|tag:<pattern> * 

232 Flattens matching keywords in all generated outputs. 

233 Matching keywords get all log messages from their 

234 child keywords and children are discarded otherwise. 

235 for: flatten for loops fully 

236 foritem: flatten individual for loop iterations 

237 name:<pattern>: flatten matched keywords using same 

238 matching rules as with 

239 `--removekeywords name:<pattern>` 

240 tag:<pattern>: flatten matched keywords using same 

241 matching rules as with 

242 `--removekeywords tag:<pattern>` 

243 --starttime timestamp Set execution start time. Timestamp must be given in 

244 format `2007-10-01 15:12:42.268` where all separators 

245 are optional (e.g. `20071001151242268` is ok too) and 

246 parts from milliseconds to hours can be omitted if 

247 they are zero (e.g. `2007-10-01`). This can be used 

248 to override start time of a single suite or to set 

249 start time for a combined suite, which would 

250 otherwise be `N/A`. 

251 --endtime timestamp Same as --starttime but for end time. If both options 

252 are used, elapsed time of the suite is calculated 

253 based on them. For combined suites, it is otherwise 

254 calculated by adding elapsed times of the combined 

255 suites together. 

256 --nostatusrc Sets the return code to zero regardless are there 

257 failures. Error codes are returned normally. 

258 --prerebotmodifier class * Class to programmatically modify the result 

259 model before creating outputs. 

260 -C --consolecolors auto|on|ansi|off Use colors on console output or not. 

261 auto: use colors when output not redirected (default) 

262 on: always use colors 

263 ansi: like `on` but use ANSI colors also on Windows 

264 off: disable colors altogether 

265 Note that colors do not work with Jython on Windows. 

266 -P --pythonpath path * Additional locations to add to the module search path 

267 that is used when importing Python based extensions. 

268 -A --argumentfile path * Text file to read more arguments from. File can have 

269 both options and output files, one per line. Contents 

270 do not need to be escaped but spaces in the beginning 

271 and end of lines are removed. Empty lines and lines 

272 starting with a hash character (#) are ignored. 

273 Example file: 

274 | --include regression 

275 | --name Regression Tests 

276 | # This is a comment line 

277 | output.xml 

278 -h -? --help Print usage instructions. 

279 --version Print version information. 

280 

281Options that are marked with an asterisk (*) can be specified multiple times. 

282For example, `--test first --test third` selects test cases with name `first` 

283and `third`. If an option accepts a value but is not marked with an asterisk, 

284the last given value has precedence. For example, `--log A.html --log B.html` 

285creates log file `B.html`. Options accepting no values can be disabled by 

286using the same option again with `no` prefix added or dropped. The last option 

287has precedence regardless of how many times options are used. For example, 

288`--merge --merge --nomerge --nostatusrc --statusrc` would not activate the 

289merge mode and would return a normal return code. 

290 

291Long option format is case-insensitive. For example, --SuiteStatLevel is 

292equivalent to but easier to read than --suitestatlevel. Long options can 

293also be shortened as long as they are unique. For example, `--logti Title` 

294works while `--lo log.html` does not because the former matches only --logtitle 

295but the latter matches both --log and --logtitle.

302ROBOT_SYSLOG_FILE Path to a file where Robot Framework writes internal

312

323

329class Rebot(RobotFramework): 

330 

331 def __init__(self): 

332 Application.__init__(self, USAGE, arg_limits=(1,), 

333 env_options='REBOT_OPTIONS', logger=LOGGER) 

334 

335 def main(self, datasources, **options): 

336 settings = RebotSettings(options) 

337 LOGGER.register_console_logger(**settings.console_output_config) 

338 if settings['Critical'] or settings['NonCritical']: 

339 LOGGER.warn("Command line options --critical and --noncritical have been " 

340 "deprecated and have no effect with Rebot. Use --skiponfailure " 

341 "when starting execution instead.") 

342 if settings['XUnitSkipNonCritical']: 

343 LOGGER.warn("Command line option --xunitskipnoncritical has been " 

344 "deprecated and has no effect.") 

345 LOGGER.disable_message_cache() 

346 rc = ResultWriter(*datasources).write_results(settings) 

347 if rc < 0: 

348 raise DataError('No outputs created.') 

349 return rc

352def rebot_cli(arguments=None, exit=True): 

353 """Command line execution entry point for post-processing outputs. 

354 

355 :param arguments: Command line options and arguments as a list of strings. 

356 Starting from RF 3.1, defaults to ``sys.argv[1:]`` if not given. 

357 :param exit: If ``True``, call ``sys.exit`` with the return code denoting 

358 execution status, otherwise just return the rc. 

359 

360 Entry point used when post-processing outputs from the command line, but 

361 can also be used by custom scripts. Especially useful if the script itself 

362 needs to accept same arguments as accepted by Rebot, because the script can 

363 just pass them forward directly along with the possible default values it 

364 sets itself. 

365 

366 Example:: 

367 

368 from robot import rebot_cli 

369 

370 rebot_cli(['--name', 'Example', '--log', 'NONE', 'o1.xml', 'o2.xml']) 

371 

372 See also the :func:`rebot` function that allows setting options as keyword 

373 arguments like ``name="Example"`` and generally has a richer API for 

374 programmatic Rebot execution. 

375 """ 

376 if arguments is None: 

377 arguments = sys.argv[1:] 

378 return Rebot().execute_cli(arguments, exit=exit)

381def rebot(*outputs, **options): 

382 """Programmatic entry point for post-processing outputs. 

383 

384 :param outputs: Paths to Robot Framework output files similarly 

385 as when running the ``rebot`` command on the command line. 

386 :param options: Options to configure processing outputs. Accepted 

387 options are mostly same as normal command line options to the ``rebot`` 

388 command. Option names match command line option long names without 

389 hyphens so that, for example, ``--name`` becomes ``name``. 

390 

391 The semantics related to passing options are exactly the same as with the 

392 :func:`~robot.run.run` function. See its documentation for more details. 

393 

394 Examples:: 

395 

396 from robot import rebot 

397 

398 rebot('path/to/output.xml') 

399 with open('stdout.txt', 'w') as stdout: 

400 rebot('o1.xml', 'o2.xml', name='Example', log=None, stdout=stdout) 

401 

402 Equivalent command line usage:: 

403 

404 rebot path/to/output.xml 

405 rebot --name Example --log NONE o1.xml o2.xml > stdout.txt 

406 """ 

407 return Rebot().execute(*outputs, **options)
