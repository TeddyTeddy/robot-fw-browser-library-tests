82 == Browsers == 

83 

84 A *browser* can be started with one of the three

89 | Browser | Browser with this engine |

107 == Contexts == 

108 

109 A *context* corresponds to set of independent incognito pages in a browser 

110 that share cookies, sessions or profile settings. Pages in two separate 

111 contexts do not share cookies, sessions or profile settings. 

112 Compared to Selenium, these do *not* require their own browser process. 

113 To get a clean environment a test can just open a new context. 

114 Due to this new independent browser sessions can be opened with 

115 Robot Framework Browser about 10 times faster than with Selenium by 

116 just opening a `New Context` within the opened browser. 

117 

118 The context layer is useful e.g. for testing different users sessions on the 

119 same webpage without opening a whole new browser context. 

120 Contexts can also have detailed configurations, such as geo-location, language settings, 

121 the viewport size or color scheme. 

122 Contexts do also support http credentials to be set, so that basic authentication 

123 can also be tested. To be able to download files within the test, 

124 the ``acceptDownloads`` argument must be set to ``True`` in `New Context` keyword. 

125 A context can contain different pages. 

126

150 = Finding elements = 

151 

152 All keywords in the library that need to interact with an element 

153 on a web page take an argument typically named ``selector`` that specifies 

154 how to find the element. 

155 

156 Selector strategies that are supported by default are listed in the table 

157 below. 

158 

159 | = Strategy = | = Match based on = | = Example = | 

160 | ``css`` | CSS selector. | ``css=.class > #login_btn`` | 

161 | ``xpath`` | XPath expression. | ``xpath=//input[@id="login_btn"]`` | 

162 | ``text`` | Browser text engine. | ``text=Login`` | 

163 | ``id`` | Element ID Attribute. | ``id=login_btn`` | 

164

166 == Explicit Selector Strategy == 

167 

168 The explicit selector strategy is specified with a prefix using syntax 

169 ``strategy=value``. Spaces around the separator are ignored, so 

170 ``css=foo``, ``css= foo`` and ``css = foo`` are all equivalent. 

171 

172 

173 == Implicit Selector Strategy == 

174 

175 *The default selector strategy is `css`.* 

176 

177 If selector does not contain one of the know explicit selector strategies, it is 

178 assumed to contain css selector. 

179 

180 Selectors that are starting with ``//`` or ``..`` are considered as xpath selectors. 

181 

182 Selectors that are in quotes are considered as text selectors. 

183 

184 Examples: 

185 

186 | # CSS selectors are default. 

187 | `Click` span > button.some_class # This is equivalent 

188 | `Click` css=span > button.some_class # to this. 

189 | 

190 | # // or .. leads to xpath selector strategy 

191 | `Click` //span/button[@class="some_class"] 

192 | `Click` xpath=//span/button[@class="some_class"] 

193 | 

194 | # "text" in quotes leads to exact text selector strategy 

195 | `Click` "Login" 

196 | `Click` text="Login" 

197 

198 

199 == CSS == 

200 

201 As written before, the default selector strategy is `css`. See 

202 [https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_Selectors | css selector] 

203 for more information. 

204 

205 Any malformed selector not starting with ``//`` or ``..`` nor starting and ending 

206 with a quote is assumed to be a css selector. 

207 

208 Example: 

209 | `Click` span > button.some_class 

210 

211 

212 == XPath == 

213 

214 XPath engine is equivalent to [https://developer.mozilla.org/en/docs/Web/API/Document/evaluate|Document.evaluate]. 

215 Example: ``xpath=//html/body//span[text()="Hello World"]``. 

216 

217 Malformed selector starting with ``//`` or ``..`` is assumed to be an xpath selector. 

218 For example, ``//html/body`` is converted to ``xpath=//html/body``. More 

219 examples are displayed in `Examples`. 

220 

221 Note that xpath does not pierce [https://developer.mozilla.org/en-US/docs/Web/Web_Components/Using_shadow_DOM|shadow_roots]. 

222 

223 

224 == Text == 

225 

226 Text engine finds an element that contains a text node with the passed text. 

227 For example, ``Click text=Login`` clicks on a login button, and 

228 ``Wait For Elements State text="lazy loaded text"`` waits for the "lazy loaded text" 

229 to appear in the page. 

230 

231 Text engine finds fields based on their labels in text inserting keywords. 

232 

233 Malformed selector starting and ending with a quote (either ``"`` or ``'``) is assumed 

234 to be a text selector. For example, ``Click "Login"`` is converted to ``Click text="Login"``. 

235 Be aware that these leads to exact matches only! 

236 More examples are displayed in `Examples`. 

237 

238 

239 === Insensitive match === 

240 

241 By default, the match is case-insensitive, ignores leading/trailing whitespace and 

242 searches for a substring. This means ``text= Login`` matches 

243 ``<button>Button loGIN (click me)</button>``. 

244 

245 === Exact match === 

246 

247 Text body can be escaped with single or double quotes for precise matching, 

248 insisting on exact match, including specified whitespace and case. 

249 This means ``text="Login "`` will only match ``<button>Login </button>`` with exactly 

250 one space after "Login". Quoted text follows the usual escaping rules, e.g. 

251 use ``\\"`` to escape double quote in a double-quoted string: ``text="foo\\"bar"``. 

252 

253 === RegEx === 

254 

255 Text body can also be a JavaScript-like regex wrapped in / symbols. 

256 This means ``text=/^hello .*!$/i`` or ``text=/^Hello .*!$/`` will match ``<span>Hello Peter Parker!</span>`` 

257 with any name after ``Hello``, ending with ``!``. 

258 The first one flagged with ``i`` for case-insensitive. 

259 See [https://regex101.com/|https://regex101.com] for more information about RegEx. 

260 

261 === Button and Submit Values === 

262 

263 Input elements of the type button and submit are rendered with their value as text, 

264 and text engine finds them. For example, ``text=Login`` matches 

265 ``<input type=button value="Login">``. 

266 

267 == Cascaded selector syntax == 

268 

269 Browser library supports the same selector strategies as the underlying 

270 Playwright node module: xpath, css, id and text. The strategy can either 

271 be explicitly specified with a prefix or the strategy can be implicit. 

272 

273 A major advantage of Browser is, that multiple selector engines can be used 

274 within one selector. It is possible to mix XPath, CSS and Text selectors while 

275 selecting a single element. 

276 

277 Selectors are strings that consists of one or more clauses separated by 

278 ``>>`` token, e.g. ``clause1 >> clause2 >> clause3``. When multiple clauses 

279 are present, next one is queried relative to the previous one's result. 

280 Browser library supports concatination of different selectors seperated by ``>>``. 

281 

282 For example: 

283 | `Highlight Elements` "Hello" >> ../.. >> .select_button 

284 | `Highlight Elements` text=Hello >> xpath=../.. >> css=.select_button 

285 

286 Each clause contains a selector engine name and selector body, e.g. 

287 ``engine=body``. Here ``engine`` is one of the supported engines (e.g. css or 

288 a custom one). Selector ``body`` follows the format of the particular engine, 

289 e.g. for css engine it should be a [https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_Selectors | css selector]. 

290 Body format is assumed to ignore leading and trailing white spaces, 

291 so that extra whitespace can be added for readability. If selector 

292 engine needs to include ``>>`` in the body, it should be escaped 

293 inside a string to not be confused with clause separator, 

294 e.g. ``text="some >> text"``. 

295 

296 Selector engine name can be prefixed with ``*`` to capture element that 

297 matches the particular clause instead of the last one. For example, 

298 ``css=article >> text=Hello`` captures the element with the text ``Hello``, 

299 and ``*css=article >> text=Hello`` (note the *) captures the article element 

300 that contains some element with the text Hello. 

301 

302 For convenience, selectors in the wrong format are heuristically converted 

303 to the right format. See `Implicit Selector Strategy` 

304

334 == Frames == 

335 

336 By default, selector chains do not cross frame boundaries. It means that a 

337 simple CSS selector is not able to select and element located inside an iframe 

338 or a frameset. For this usecase, there is a special selector ``>>>`` which can 

339 be used to combine a selector for the frame and a selector for an element 

340 inside a frame. 

341 

342 Given this simple pseudo html snippet: 

343 | <iframe id="iframe" src="src.html"> 

344 | #document 

345 | <!DOCTYPE html> 

346 | <html> 

347 | <head></head> 

348 | <body> 

349 | <button id="btn">Click Me</button> 

350 | </body> 

351 | </html> 

352 | </iframe> 

353 

354 Here's a keyword call that clicks the button inside the frame. 

355 

356 | Click id=iframe >>> id=btn 

357 

358 The selectors on the left and right side of ``>>>`` can be any valid selectors. 

359 The selector clause directly before the frame opener ``>>>`` must select the frame element.

361 == WebComponents and Shadow DOM == 

362 

363 Playwright and so also Browser are able to do automatic piercing of Shadow DOMs 

364 and therefore are the best automation technology when working with WebComponents. 

365 

366 Also other technologies claim that they can handle 

367 [https://developer.mozilla.org/en-US/docs/Web/Web_Components/Using_shadow_DOM|Shadow DOM and Web Components]. 

368 However, non of them do pierce shadow roots automatically, 

369 which may be inconvenient when working with Shadow DOM and Web Components. 

370 

371 For that reason, css engine pierces shadow roots. More specifically, every 

372 [https://developer.mozilla.org/en-US/docs/Web/CSS/Descendant_combinator|Descendant combinator] 

373 pierces an arbitrary number of open shadow roots, including the implicit descendant combinator 

374 at the start of the selector. 

375 

376 That means, it is not nessesary to select each shadow host, open its shadow root and 

377 select the next shadow host until you reach the element that should be controlled. 

378 

379 === CSS:light === 

380 

381 ``css:light`` engine is equivalent to [https://developer.mozilla.org/en/docs/Web/API/Document/querySelector | Document.querySelector] 

382 and behaves according to the CSS spec. 

383 However, it does not pierce shadow roots. 

384 

385 ``css`` engine first searches for elements in the light dom in the iteration order, 

386 and then recursively inside open shadow roots in the iteration order. It does not 

387 search inside closed shadow roots or iframes. 

388 

389 Examples: 

390 

391 | <article> 

392 | <div>In the light dom</div> 

393 | <div slot='myslot'>In the light dom, but goes into the shadow slot</div> 

394 | <open mode shadow root> 

395 | <div class='in-the-shadow'> 

396 | <span class='content'> 

397 | In the shadow dom 

398 | <open mode shadow root> 

399 | <li id='target'>Deep in the shadow</li> 

400 | </open mode shadow root> 

401 | </span> 

402 | </div> 

403 | <slot name='myslot'></slot> 

404 | </open mode shadow root> 

405 | </article> 

406 

407 Note that ``<open mode shadow root>`` is not an html element, but rather a shadow root 

408 created with ``element.attachShadow({mode: 'open'})``. 

409 

410 - Both ``"css=article div"`` and ``"css:light=article div"`` match the first ``<div>In the light dom</div>``. 

411 - Both ``"css=article > div"`` and ``"css:light=article > div"`` match two ``div`` elements that are direct children of the ``article``. 

412 - ``"css=article .in-the-shadow"`` matches the ``<div class='in-the-shadow'>``, piercing the shadow root, while ``"css:light=article .in-the-shadow"`` does not match anything. 

413 - ``"css:light=article div > span"`` does not match anything, because both light-dom ``div`` elements do not contain a ``span``. 

414 - ``"css=article div > span"`` matches the ``<span class='content'>``, piercing the shadow root. 

415 - ``"css=article > .in-the-shadow"`` does not match anything, because ``<div class='in-the-shadow'>`` is not a direct child of ``article`` 

416 - ``"css:light=article > .in-the-shadow"`` does not match anything. 

417 - ``"css=article li#target"`` matches the ``<li id='target'>Deep in the shadow</li>``, piercing two shadow roots. 

418 

419 === text:light === 

420 

421 ``text`` engine open pierces shadow roots similarly to ``css``, while ``text:light`` does not. 

422 Text engine first searches for elements in the light dom in the iteration order, and then 

423 recursively inside open shadow roots in the iteration order. It does not search inside 

424 closed shadow roots or iframes. 

425 

426 === id, data-testid, data-test-id, data-test and their :light counterparts === 

427 

428 Attribute engines are selecting based on the corresponding attribute value. 

429 For example: ``data-test-id=foo`` is equivalent to ``css=[data-test-id="foo"]``, 

430 and ``id:light=foo`` is equivalent to ``css:light=[id="foo"]``.

443 = Assertions = 

444 

445 Keywords that accept arguments ``assertion_operator`` <`AssertionOperator`> and ``assertion_expected`` 

446 can optionally assert. 

447 %ASSERTION_TABLE% 

448 

449 But default the keywords will provide an error message if the assertion fails, 

450 but default error message can be overwritten with a ``message`` argument. The 

451 ``message`` argument accepts `{value}`, `{value_type}`, `{expected}` and 

452 `{expected_type}` [https://docs.python.org/3/library/stdtypes.html#str.format|format] 

453 options. The `{value}` is the value returned by the keyword and the `{expected}` 

454 is the expected value defined by the user, usually value in the 

455 ``assertion_expected`` argument. The `{value_type}` and 

456 `{expected_type}` are the type definitions from `{value}` and `{expected}` 

457 arguments. In similar fashion as Python 

458 [https://docs.python.org/3/library/functions.html#type|type] returns type definition. 

459 Assertions will retry until ``timeout`` has expired if they do not pass. 

460 

461 The assertion ``assertion_expected`` value is not converted by the library and 

462 is used as is. Therefore when assertion is made, the ``assertion_expected`` 

463 argument value and value returned the keyword must have same type. If types 

464 are not same, assertion will fail. Example `Get Text` always returns a string 

465 and has to be compared with a string, even the returnd value might look like 

466 a number. 

467 

468 Other Keywords have other specific types they return. 

469 `Get Element Count` always returns an integer. 

470 `Get Bounding Box` and `Get Viewport Size` can be filtered. 

471 They return a dictionary without filter and a number when filtered. 

472 These Keywords do autoconvert the expected value if a number is returned. 

473 

474 * < less or greater > With Strings* 

475 Compairisons of strings with ``greater than`` or ``less than`` compares each character, 

476 starting from 0 reagarding where it stands in the code page. 

477 Example: ``A < Z``, ``Z < a``, ``ac < dc` 

478 It does never compare the length of elements. Neither lists nor strings. 

479 The comparison stops at the first character that is different. 

480 Examples: ``'abcde' < 'abd'``, ``'100.000' < '2'`` 

481 In Python 3 and therefore also in Browser it is not possible to compare numbers 

482 with strings with a greater or less operator. 

483 On keywords that return numbers, the given expected value is automatically 

484 converted to a number before comparison. 

485 

486 

487 The getters `Get Page State` and `Get Browser Catalog` return a dictionary. Values of the dictionary can directly asserted. 

488 Pay attention of possible types because they are evaluated in Python. For example: 

489 

490 | Get Page State validate 2020 >= value['year'] # Compairsion of numbers 

491 | Get Page State validate "IMPORTANT MESSAGE!" == value['message'] # Compairsion of strings 

492

499 For example ``Get Title then 'TITLE: '+value``.

503 for more info on the syntax.

620 Configure context and page automatic closing. Default is ``TEST``, 

621 for more details, see `AutoClosingLevel` 

622 - ``retry_assertions_for`` <str> 

623 Timeout for retrying assertions on keywords before failing the keywords. 

624 This timeout starts counting from the first failure.

638 - ``enable_presenter_mode`` <bool>

769 def _prune_execution_stack(self, catalog_before: dict) -> None: 

770 catalog_after = self.get_browser_catalog() 

771 ctx_before_ids = [c["id"] for b in catalog_before for c in b["contexts"]] 

772 ctx_after_ids = [c["id"] for b in catalog_after for c in b["contexts"]] 

773 new_ctx_ids = [c for c in ctx_after_ids if c not in ctx_before_ids] 

774 for ctx_id in new_ctx_ids: 

775 self._playwright_state.switch_context(ctx_id) 

776 self._playwright_state.close_context() 

777 pages_before = [ 

778 (p["id"], c["id"]) 

779 for b in catalog_before 

780 for c in b["contexts"] 

781 for p in c["pages"] 

782 ] 

783 pages_after = [ 

784 (p["id"], c["id"]) 

785 for b in catalog_after 

786 for c in b["contexts"] 

787 for p in c["pages"] 

788 if c["id"] not in new_ctx_ids 

789 ] 

790 new_page_ids = [p for p in pages_after if p not in pages_before] 

791 for page_id, ctx_id in new_page_ids: 

792 self._playwright_state.close_page(page_id, ctx_id)

808 def start_keyword(self, name, attrs): 

809 """Take screenshot of tests that have failed due to timeout. 

810 

811 This method is part of the Listener API implemented by the library. 

812 

813 This can be done with BuiltIn keyword `Run Keyword If Timeout 

814 Occurred`, but the problem there is that you have to remember to 

815 put it into your Suite/Test Teardown. Since taking screenshot is 

816 the most obvious thing to do on failure, let's do it automatically. 

817 

818 This cannot be implemented as a `end_test` listener method, since at 

819 that time, the teardown has already been executed and browser may have 

820 been closed already. This implementation will take the screenshot 

821 before the teardown begins to execute. 

822 """ 

823 self.current_arguments = tuple(attrs["args"]) 

824 if attrs["type"] == "Teardown": 

825 timeout_pattern = "Test timeout .* exceeded." 

826 test = EXECUTION_CONTEXTS.current.test 

827 if ( 

828 test is not None 

829 and test.status == "FAIL" 

830 and re.match(timeout_pattern, test.message) 

831 ): 

832 self.screenshot_on_failure(test.name)

834 def keyword_error(self): 

835 """Sends screenshot command to Playwright. 

836 

837 Only works during testing since this uses robot's outputdir for output. 

838 """ 

839 if self._running_on_failure_keyword or not self.run_on_failure_keyword: 

840 return 

841 try: 

842 self._running_on_failure_keyword = True 

843 if is_same_keyword(self.run_on_failure_keyword, "Take Screenshot"): 

844 self.take_screenshot(self._failure_screenshot_path()) 

845 else: 

846 BuiltIn().run_keyword(self.run_on_failure_keyword) 

847 except Exception as err: 

848 logger.warn( 

849 f"Keyword '{self.run_on_failure_keyword}' could not be run on failure:\n{err}" 

850 ) 

851 finally: 

852 self._running_on_failure_keyword = False
