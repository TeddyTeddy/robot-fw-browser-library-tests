57Robot Framework is a generic open source automation framework for acceptance 

58testing, acceptance test-driven development (ATDD) and robotic process 

59automation (RPA). It has simple, easy-to-use syntax that utilizes the 

60keyword-driven automation approach. Keywords adding new capabilities are 

61implemented in libraries using either Python or Java. New higher level 

62keywords can also be created using Robot Framework's own syntax. 

63 

64The easiest way to execute Robot Framework is using the `robot` command created 

65as part of the normal installation. Alternatively it is possible to execute 

66the `robot` module directly like `python -m robot`, where `python` can be 

67replaced with any supported Python interpreter such as `jython`, `ipy` or 

68`python3`. Yet another alternative is running the `robot` directory like 

69`python path/to/robot`. Finally, there is a standalone JAR distribution 

70available. 

71 

72Tests (or tasks in RPA terminology) are created in files typically having the 

73`*.robot` extension. Files automatically create test (or task) suites and 

74directories with these files create higher level suites. When Robot Framework 

75is executed, paths to these files or directories are given to it as arguments. 

76 

77By default Robot Framework creates an XML output file and a log and a report in 

78HTML format, but this can be configured using various options listed below. 

79Outputs in HTML format are for human consumption and XML output for integration 

80with other systems. XML outputs can also be combined and otherwise further 

81post-processed with the Rebot tool that is an integral part of Robot Framework. 

82Run `rebot --help` for more information. 

83 

84Robot Framework is open source software released under Apache License 2.0. 

85For more information about the framework and the rich ecosystem around it 

86see http://robotframework.org/.

107 documentation contains spaces, it must be quoted. 

108 Example: --doc "Very *good* example"

111 Example: --metadata Version:1.2

113 -t --test name * Select tests by name or by long name containing also

246 --expandkeywords tag:expand 

247 New in RF 3.2. 

248 --removekeywords all|passed|for|wuks|name:<pattern>|tag:<pattern> *

250 Keywords containing warnings are not removed except

252 all: remove data from all keywords 

253 passed: remove data only from keywords in passed

255 for: remove passed iterations from for loops 

256 wuks: remove all but the last failing keyword 

257 inside `BuiltIn.Wait Until Keyword Succeeds` 

258 name:<pattern>: remove data from keywords that match 

259 the given pattern. The pattern is matched 

260 against the full name of the keyword (e.g. 

261 'MyLib.Keyword', 'resource.Second Keyword'), 

262 is case, space, and underscore insensitive, 

263 and may contain `*`, `?` and `[]` wildcards. 

264 Examples: --removekeywords name:Lib.HugeKw 

265 --removekeywords name:myresource.* 

266 tag:<pattern>: remove data from keywords that match 

267 the given pattern. Tags are case and space 

268 insensitive and patterns can contain `*`, 

269 `?` and `[]` wildcards. Tags and patterns 

270 can also be combined together with `AND`, 

271 `OR`, and `NOT` operators. 

272 Examples: --removekeywords foo 

273 --removekeywords fooANDbar*

275 Flattens matching keywords in the generated log file. 

276 Matching keywords get all log messages from their 

277 child keywords and children are discarded otherwise. 

278 for: flatten for loops fully 

279 foritem: flatten individual for loop iterations 

280 name:<pattern>: flatten matched keywords using same 

281 matching rules as with

283 tag:<pattern>: flatten matched keywords using same 

284 matching rules as with

333 -P --pythonpath path * Additional locations (directories, ZIPs, JARs) where 

334 to search test libraries and other extensions when 

335 they are imported. Multiple paths can be given by 

336 separating them with a colon (`:`) or by using this 

337 option several times. Given path can also be a glob 

338 pattern matching multiple paths. 

339 Examples: 

340 --pythonpath libs/ --pythonpath resources/*.jar

342 -A --argumentfile path * Text file to read more arguments from. Use special 

343 path `STDIN` to read contents from the standard input 

344 stream. File can have both options and input files 

345 or directories, one per line. Contents do not need to 

346 be escaped but spaces in the beginning and end of

360Options that are marked with an asterisk (*) can be specified multiple times. 

361For example, `--test first --test third` selects test cases with name `first` 

362and `third`. If an option accepts a value but is not marked with an asterisk, 

363the last given value has precedence. For example, `--log A.html --log B.html`

365using the same option again with `no` prefix added or dropped. The last option 

366has precedence regardless of how many times options are used. For example, 

367`--dryrun --dryrun --nodryrun --nostatusrc --statusrc` would not activate the 

368dry-run mode and would return a normal return code. 

369

371equivalent to but easier to read than --suitestatlevel. Long options can 

372also be shortened as long as they are unique. For example, `--logti Title` 

373works while `--lo log.html` does not because the former matches only --logtitle 

374but the latter matches --log, --loglevel and --logtitle.

417class RobotFramework(Application): 

418 

419 def __init__(self): 

420 Application.__init__(self, USAGE, arg_limits=(1,), 

421 env_options='ROBOT_OPTIONS', logger=LOGGER) 

422 

423 def main(self, datasources, **options): 

424 settings = RobotSettings(options) 

425 LOGGER.register_console_logger(**settings.console_output_config) 

426 if settings['Critical'] or settings['NonCritical']: 

427 LOGGER.warn("Command line options --critical and --noncritical have been " 

428 "deprecated. Use --skiponfailure instead.") 

429 if settings['XUnitSkipNonCritical']: 

430 LOGGER.warn("Command line option --xunitskipnoncritical has been " 

431 "deprecated and has no effect.") 

432 LOGGER.info('Settings:\n%s' % unic(settings)) 

433 builder = TestSuiteBuilder(settings['SuiteNames'], 

434 included_extensions=settings.extension, 

435 rpa=settings.rpa, 

436 allow_empty_suite=settings.run_empty_suite) 

437 suite = builder.build(*datasources) 

438 settings.rpa = suite.rpa 

439 if settings.pre_run_modifiers: 

440 suite.visit(ModelModifier(settings.pre_run_modifiers, 

441 settings.run_empty_suite, LOGGER)) 

442 suite.configure(**settings.suite_config) 

443 with pyloggingconf.robot_handler_enabled(settings.log_level): 

444 old_max_error_lines = text.MAX_ERROR_LINES 

445 text.MAX_ERROR_LINES = settings.max_error_lines 

446 try: 

447 result = suite.run(settings) 

448 finally: 

449 text.MAX_ERROR_LINES = old_max_error_lines 

450 LOGGER.info("Tests execution ended. Statistics:\n%s" 

451 % result.suite.stat_message) 

452 if settings.log or settings.report or settings.xunit: 

453 writer = ResultWriter(settings.output if settings.log 

454 else result) 

455 writer.write_results(settings.get_rebot_settings()) 

456 return result.return_code

458 def validate(self, options, arguments): 

459 return self._filter_options_without_value(options), arguments

462 return dict((name, value) for name, value in options.items()

466def run_cli(arguments=None, exit=True): 

467 """Command line execution entry point for running tests. 

468 

469 :param arguments: Command line options and arguments as a list of strings. 

470 Starting from RF 3.1, defaults to ``sys.argv[1:]`` if not given. 

471 :param exit: If ``True``, call ``sys.exit`` with the return code denoting 

472 execution status, otherwise just return the rc. 

473 

474 Entry point used when running tests from the command line, but can also 

475 be used by custom scripts that execute tests. Especially useful if the 

476 script itself needs to accept same arguments as accepted by Robot Framework, 

477 because the script can just pass them forward directly along with the 

478 possible default values it sets itself. 

479 

480 Example:: 

481 

482 from robot import run_cli 

483 

484 # Run tests and return the return code. 

485 rc = run_cli(['--name', 'Example', 'tests.robot'], exit=False) 

486 

487 # Run tests and exit to the system automatically. 

488 run_cli(['--name', 'Example', 'tests.robot']) 

489 

490 See also the :func:`run` function that allows setting options as keyword 

491 arguments like ``name="Example"`` and generally has a richer API for 

492 programmatic test execution. 

493 """ 

494 if arguments is None: 

495 arguments = sys.argv[1:] 

496 return RobotFramework().execute_cli(arguments, exit=exit)

499def run(*tests, **options): 

500 """Programmatic entry point for running tests. 

501 

502 :param tests: Paths to test case files/directories to be executed similarly 

503 as when running the ``robot`` command on the command line. 

504 :param options: Options to configure and control execution. Accepted 

505 options are mostly same as normal command line options to the ``robot`` 

506 command. Option names match command line option long names without 

507 hyphens so that, for example, ``--name`` becomes ``name``. 

508 

509 Most options that can be given from the command line work. An exception 

510 is that options ``--pythonpath``, ``--argumentfile``, ``--help`` and 

511 ``--version`` are not supported. 

512 

513 Options that can be given on the command line multiple times can be 

514 passed as lists. For example, ``include=['tag1', 'tag2']`` is equivalent 

515 to ``--include tag1 --include tag2``. If such options are used only once, 

516 they can be given also as a single string like ``include='tag'``. 

517 

518 Options that accept no value can be given as Booleans. For example, 

519 ``dryrun=True`` is same as using the ``--dryrun`` option. 

520 

521 Options that accept string ``NONE`` as a special value can also be used 

522 with Python ``None``. For example, using ``log=None`` is equivalent to 

523 ``--log NONE``. 

524 

525 ``listener``, ``prerunmodifier`` and ``prerebotmodifier`` options allow 

526 passing values as Python objects in addition to module names these command 

527 line options support. For example, ``run('tests', listener=MyListener())``. 

528 

529 To capture the standard output and error streams, pass an open file or 

530 file-like object as special keyword arguments ``stdout`` and ``stderr``, 

531 respectively. 

532 

533 A return code is returned similarly as when running on the command line. 

534 Zero means that tests were executed and no critical test failed, values up 

535 to 250 denote the number of failed critical tests, and values between 

536 251-255 are for other statuses documented in the Robot Framework User Guide. 

537 

538 Example:: 

539 

540 from robot import run 

541 

542 run('path/to/tests.robot') 

543 run('tests.robot', include=['tag1', 'tag2'], splitlog=True) 

544 with open('stdout.txt', 'w') as stdout: 

545 run('t1.robot', 't2.robot', name='Example', log=None, stdout=stdout) 

546 

547 Equivalent command line usage:: 

548 

549 robot path/to/tests.robot 

550 robot --include tag1 --include tag2 --splitlog tests.robot 

551 robot --name Example --log NONE t1.robot t2.robot > stdout.txt 

552 """ 

553 return RobotFramework().execute(*tests, **options)
